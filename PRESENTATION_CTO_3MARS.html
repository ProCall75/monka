Un simulateur visuel rapide pour que Benjamin et l'équipe puissent voir le moteur clinique fonctionner.

En résumé : L'app ne va pas en production — la logique du moteur, si. Le passage en prod = votre architecture + notre vélocité.

🧠 Concepts Clés

Les règles fondamentales du Kernel

Ce que le moteur fait et ne fait pas — les concepts qu'on maîtrise

🔴 CCC — Condition Critique Composite

Combinaison de signaux faibles de vulnérabilités différentes → alerte de niveau supérieur. Différenciateur absolu de Monka.

🏆 ASR — Action Seuil de Réussite

Objectif mesurable par MP. Validée quand toutes les MT contributives (📍 STRUC/SEC/MED) sont complétées.

K3 — Englobement

Si une catégorie est activée à plusieurs niveaux → seul le niveau le plus haut s'affiche. 🔴 absorbe 🟠 absorbe 🟢.

K13 — Scoring indépendant

Le score mesure l'intensité. Il ne déclenche jamais un MP. C'est un indicateur, pas un prescripteur.

Double wording (K6) : Chaque reco et MT existe en 2 versions — Aidant (bienveillant) et IDEC (directif professionnel). Les mêmes actions, un langage adapté.

🎯 Gap Analysis

Ce qui manque pour la production — la vraie question

Le simulateur actuel n'ira pas en prod. La question : Greenfield ou Brownfield ?

🆕 Greenfield (nouvelle app from scratch)

On repart de zéro avec l'architecture CTO. Le Kernel est extrait comme module pur. Le reste est neuf.

→ Cas probable pour MyMonka (app aidant) et potentiellement pour CRM Lifeline

🔧 Brownfield (intégrer dans l'existant)

Si Monka a déjà une codebase CRM → on intègre le Kernel dans leur stack. Audit de l'existant nécessaire.

→ À préciser avec le CTO : quelle est l'infra existante ?

📋 Prérequis production identifiés (dans les 2 cas)

❌ Tests Unitaires + intégration + E2E
❌ CI/CD Pipeline avec quality gates
❌ Auth + RLS Gestion utilisateurs, permissions
❌ Monitoring Sentry + alerting + logging structuré
❌ Error Handling Error Boundaries + graceful degradation
❌ Docker prod Multi-stage build, images versionnées
❌ Sécurité / HDS Conformité données de santé
❌ Env vars Secrets management, 0 clé en dur
❌ Environnements Dev / Staging / Prod isolés
❌ Documentation API docs, runbooks, ADR
C'est précisément pourquoi le CTO est essentiel : il définit le cadre (stack, conventions, sécurité, infra) — on exécute dans ce cadre avec notre connaissance du produit.

🏗️ Architecture

Options d'architecture — à trancher ensemble

Plusieurs approches possibles selon l'existant et la vision technique — le Kernel s'adapte à toutes

🗂️ Monorepo

Un seul repo, tout centralisé (Turborepo, Nx)

✅ Partage de code natif
✅ Refacto atomique
⚠️ CI/CD plus complexe

📦 Multi-repo

Repos séparés, Kernel en package NPM

✅ Isolation forte
✅ Équipes autonomes
⚠️ Versioning à gérer

🔀 Hybride

Kernel en module local, apps séparées

✅ Flexibilité max
✅ Migration progressive
⚠️ Conventions à formaliser

🧩 Stack possible (à valider)

Mobile React Native / Flutter / natif
Web React, Vue, ou Next.js
Backend À auditer (existant Monka ?)
DB PostgreSQL sur infra HDS certifiée
Auth À définir selon le contexte
Infra Docker + CI/CD (conventions CTO)
Le Kernel s'adapte à toutes ces options — c'est du TypeScript pur, zéro dépendance. Quelle que soit l'architecture retenue, la logique clinique reste identique.

🔧 Workflow

Comment on garantit la qualité du code livré

Un processus structuré, adaptable aux standards du CTO, avec vérification à chaque étape

📋 SOPs formalisés

Chaque tâche suit un Standard Operating Procedure : scope clair, critères d'acceptance, checklist de fin. Adaptés aux conventions du CTO dès le jour 1.

🤖 Maîtrise du contexte IA

L'IA est un outil, pas un pilote. On gère le contexte, les prompts, la vérification. Chaque output est relu et validé avant commit.

✅ Certification par checklist

Avant chaque PR : checklist complète (lint, types, tests, build, review manuelle). Rien ne part sans validation.

Le cycle pour chaque ticket

1. Lecture du ticket + contexte métier
2. Implémentation (dev assisté IA + vérification systématique)
3. Vérification checklist (lint, types, tests, build)
4. PR claire avec description + screenshots si UI
5. Review CTO → corrections → merge

Ce que le CTO contrôle

Conventions Nommage, structure, patterns → on suit
CI/CD Pipeline, Docker, déploiement → son choix
Tests Framework, couverture min → on applique
Architecture Monorepo, packages, API → sa décision
